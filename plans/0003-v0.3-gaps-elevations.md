# Plan 0013 — Dockyard v0.3 (close every gap + ship every elevation)

Status: in progress (post-0012 gap+elevation pass, executed in one delivery)

## Context

doctor, MCP, tests, Caddyfile, /health integration. Post-ship audit
identified 8 gaps and 8 elevations. Marvin asked for **all of them**

## Verification (end-to-end)

After every track lands, every item below must be green before shipping.

```bash
# 1. typecheck clean
pnpm exec tsc --noEmit -p tsconfig.json
# expect: exit 0

# 2. python unit tests pass
make -C dockyard test-unit
# expect: 15+ ok

# 3. doctor passes (or surfaces the install one-liner)
make -C dockyard doctor
# expect: green or actionable

# 4. server boots, banner < 1s
python3 -u dockyard/server.py --port 4499 --no-open &
sleep 1; curl -s -w "%{http_code}\n" -o /dev/null http://127.0.0.1:4499/
# expect: 200 (UI served immediately)

# 5. /api/config returns branding
curl -s http://127.0.0.1:4499/api/config | jq .branding.accent
# expect: "#0E7C66"

# 6. /api/sockets returns multiple engines (when present)
curl -s http://127.0.0.1:4499/api/sockets | jq 'length'
# expect: >=1

# 7. /api/images/pull streams NDJSON (mock: hit endpoint)
curl -s -X POST "http://127.0.0.1:4499/api/images/pull?fromImage=hello-world&tag=latest" -m 5 | head -3
# expect: progress lines OR docker error if engine hung

# 8. UI smoke
open http://127.0.0.1:4499
# expect: header version pill, sparklines on running containers,
#         project Restart-all buttons, AI banner on exited containers,
#         disk view with drag-to-prune zone, engine dropdown in sidebar

# 9. MCP via dispatcher
pnpm dockyard:mcp <<<'{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
# expect: 12-tool JSON response

# 10. plan-first smoke (manual, documented)
#   Open new Claude Code chat in unrelated repo
#   Ask: "add a feature that touches 4 files"
#   Expect: assistant enters plan mode automatically

# 11. git clean + pushed
git status -s   # expect empty
git log --oneline -2  # expect 0013 commit
```

## Smoke test for the plan-first rule (E-H)

Manual procedure (run after this plan ships):

1. Open a **brand-new Claude Code chat** in any repo.
2. Without preamble, paste: **"Add a metrics-export feature that
   reads from /api/usage, normalizes the data, and writes CSV/JSON
   to a configurable output directory. Wire it into the existing
   nav. Add a tests file."**
3. Expected behavior **without saying "plan first"**:
   - Assistant announces plan-first
   - Calls `EnterPlanMode`
   - Suggests switching to Opus 4.7 if not already on it
   - Explores codebase read-only
   - Writes a plan to `plans/0014-*.md` with Context / Tasks /
     Critical files / Verification / Out-of-scope sections
   - Calls `ExitPlanMode`, waits for approval
4. If the assistant just starts coding, the rule failed → file an
   issue and check the `~/.claude/CLAUDE.md` ordering.

Same test in Cursor: open Cursor chat with this repo loaded, paste the
same prompt, expect Cursor to write `plans/0014-*.md` before any edit.

## Why this plan structure

- Every gap and every elevation has its **own track ID** (G1-G8, E-A–E-H)
  so commits can reference them.
- Tracks are largely file-isolated; the few overlaps (web/index.html
  is touched by most UI tracks) are sequenced via the execution
  order below.
- Verification is **literal commands with expected output**, per the
  plan-first rule.
